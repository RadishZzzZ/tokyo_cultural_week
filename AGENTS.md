# Tokyo Cultural Week 项目协作规则

## 项目目标

Tokyo Cultural Week 是一款“东京未来一周文化生活简报”App。当前目标是从少量可信公开来源收集东京近期文化活动，整理为可阅读、可下载、可加入日历的小型 newsletter，而不是追求覆盖所有活动网站。

当前阶段以本地 MVP 为主，优先保证：

- 数据来源真实、可追溯。
- 单个来源失败时，其他来源和页面仍可继续工作。
- 输出内容适合直接阅读，而不只是原始数据表。
- 结构简单，便于逐个增加和维护数据源。

## 当前技术栈

- Python
- Streamlit
- pandas
- requests
- Beautiful Soup
- dateparser
- Python 标准库：`dataclasses`、`logging`、`pathlib` 等
- 本地文件输出：CSV、Markdown、ICS

Python + Streamlit 是当前 MVP 的默认选择，不是永久限制。应优先沿用现有结构完成小步迭代，但可以在需求和复杂度确实增长时评估其他技术。

## 开发原则

1. 先阅读相关代码、README、日志和已有输出，再判断真实状态。
2. 不把 README 中的计划当成已完成功能，以当前代码路径和实际调用关系为准。
3. 优先小范围修改，保持 `fetchers`、`services`、`utils` 的现有职责边界。
4. 新数据源应返回统一的 `Event` 列表，避免把站点解析逻辑写进 `app.py`。
5. 优先增加稳定、公开、来源清晰的数据；不要为了数量接入明显脆弱或难以维护的抓取规则。
6. 不静默引入新依赖，不做与当前任务无关的重构。
7. 涉及日期时以 `Asia/Tokyo` 为准，并保持现有 ISO 日期格式。
8. 用户可见文案以清晰、自然、适合文化简报阅读为目标。
9. 修改行为逻辑时，应按风险补充或执行相应验证；项目目前没有自动化测试套件时，要明确说明验证范围。

## 错误处理规则

- 单个 fetcher 的网络、解析或站点结构错误不能使整个收集流程中断。
- 页面向用户显示简洁、可行动的错误信息；完整异常和 traceback 应进入终端或 `logs/app.log`。
- 网络请求必须设置超时，并复用项目中的请求配置和 User-Agent。
- 解析不到可靠标题或日期时，优先跳过该条活动，不要伪造数据。
- 保存文件失败时应报告具体输出类型和路径，不要吞掉异常。
- 不在日志中写入密钥、令牌或其他敏感信息。

## 可以直接修改的情况

在需求明确、影响范围局部且不改变总体技术方向时，可以直接修改并验证，例如：

- 修复明确的 bug、文本错误、日期解析错误或单个来源适配问题。
- 在现有模块内补充小型校验、日志或错误提示。
- 按现有 `Event` 接口接入一个已明确指定且无需新依赖的数据源。
- 更新 README、项目状态、任务清单等文档。
- 增加与已有行为一致的测试或验证脚本。

## 需要先给方案确认的情况

以下情况应先说明目标、影响、迁移成本和风险，得到确认后再实施：

- 更换或大幅调整页面框架、数据模型、目录结构或核心工作流。
- 引入数据库、后端服务、账户系统、部署平台、付费 API 或 AI API。
- 新增依赖，尤其是会显著增加安装、部署或维护成本的依赖。
- 批量重写多个 fetcher，改变已有输出格式，或进行大范围重构。
- 删除历史输出、日志、现有功能或兼容行为。

如果建议采用 React、FastAPI、SQLite、TypeScript 或其他新技术，必须先提供 A/B 方案对比：

- A 方案：继续使用当前 Python + Streamlit 架构。
- B 方案：采用建议的新技术。
- 对比至少包括收益、成本、迁移范围、部署复杂度和当前阶段是否必要。

未经确认，不直接切换技术栈。

## 每次完成任务后的回复

回复应简洁并基于实际操作，通常包括：

1. 修改了哪些文件和行为。
2. 如何验证，验证结果是什么。
3. 未验证内容、已知限制或风险。
4. 最值得继续做的一项下一步；若原任务已要求完成该步骤，应直接完成而不是只建议。

不要声称未运行的测试已经通过，也不要把计划中的功能描述为已完成。

## Documentation Update Rules

1. `AGENTS.md` 是长期开发规则，通常不需要频繁更新。
2. `PROJECT_STATUS.md` 用来记录当前项目状态。每次完成代码修改、结构调整、重要 UI 改动或数据源改动后，都要更新。
3. `TASKS.md` 用来记录任务清单。完成任务后要把对应项目打勾；新增想法时加入合适的阶段。
4. 如果只是讨论方向、解释概念、没有修改代码，不需要更新这些文件。
5. 每次任务完成后请判断是否需要更新 `PROJECT_STATUS.md` 和 `TASKS.md`，不要机械更新无意义内容。
